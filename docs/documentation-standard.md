# 🏛️ Skill-Manager ドキュメント標準 (DOCUMENTATION_STANDARD)

このドキュメントは、Skill-Manager プロジェクトにおけるドキュメント作成の階層構造、命名規則、および記述スタイルを定義します。人間と AI エージェントの双方が、情報の配置場所を即座に判断し、情報の重複を排除することを目的とします。

---

## 1. はじめに (Purpose)
プロジェクトの規模が拡大し、複数の AI エージェントが関与するようになると、情報の断片化と重複が最大の敵となります。本標準は、各ドキュメントに明確な「情報の主権（Ownership）」を与えることで、メンテナンス性を高め、エージェントが迷わずに正しい情報を取得・更新できる環境を構築します。

## 2. ドキュメント戦略: 役割ベースの分類 (Role-based Strategy)

Skill-Manager では、情報の「目的」と「対象読者」に応じて、ドキュメントを以下の役割で分類します。

| 分類 (Role) | 対象ファイル | 役割の定義 / 記述内容の厳守ルール |
| :--- | :--- | :--- |
| **入口 (Portal)** | `README.md` | **ユーザー向け**: 5分で全体像（ビジョン・概念図・最小セットアップ）を理解させ、他ドキュメントへ誘導する。詳細な仕様は書かない。 |
| **価値 (Value)** | `FEATURES.md` | **価値の全体像**: Mission/Vision を体現する User Story Map。実装済み・未実装を問わず、全ての機能の進捗を管理する。 |
| **設計 (Logic)** | `docs/*.md` | **開発者向け（詳細）**: システムの内側、ディレクトリ構造、プロトコル等の「なぜ・どう作られているか」の設計図。 |
| **規約 (Rules)** | `PROJECT_RULES.md` | **開発者・AI向け**: 開発時の振る舞い、スコープ、禁止事項、命名規則等の「開発の憲法（SSoT）」。 |
| **動態 (Live)** | `todos.md` | **実行タスク**: 直近の完了項目と、現在進行中・未着手の「具体的な作業単位」のみを扱う。 |

---

## 3. ファイル命名規則と配置 (Naming & Directory)

-   **命名規則**: 
    -   ファイル名はすべて **kebab-case** (`document-standard.md`) を使用してください。
    -   日本語版を作成する場合は、ファイル名の末尾に `_ja.md` を付与します。
-   **ディレクトリ構造**:
    -   プロジェクト全体に関わる戦略的ドキュメントはルートディレクトリに配置します。
    -   特定のトピックに関する詳細な技術ドキュメントは `docs/` 配下に集約します。
    -   `docs/` 配下のファイルが 200 行を超える場合、または概念的に独立している場合は、積極的にファイルを分割してください。
-   **日英併記**: 
    -   当面は **日本語優先** でメンテナンスします。
    -   主要なドキュメント（README, PROJECT_RULES 等）は、必ず日本語版 (`_ja.md`) を作成・維持してください。

## 4. 記述スタイル (Language & Style)

AI が文脈を誤認せず、人間が素早く読めるよう、以下のルールを遵守してください。

-   **平易かつ簡潔**: 詩的な表現や過度な装飾を避け、直接的な説明を心がけてください。
-   **比喩の限定的利用**: 「ハブとターゲット」「架け橋（Bridge）」など、システムの概念理解を助けるために不可欠な場合のみ比喩を使用してください。
-   **スキャナビリティ**: 箇条書き（Bullet points）やテーブルを活用し、一目で構造がわかるようにしてください。
-   **一貫性**: 確立された用語（例：「実体」「ソース」等）を使い、類義語を乱立させないでください。

## 5. 相互参照のルール (References)

-   **相対パスの徹底**: 内部リンクには常に相対パス (`../docs/xxx.md`) を使用してください。
-   **情報の重複排除**: あるドキュメントに詳細が書かれている場合、他ではその詳細を繰り返さず、リンクのみを貼ってください。
-   **リンクの検証**: 新しいドキュメントを追加した際は、関連するポータル（README や Content Map）からのリンクを更新してください。

## 6. ライフサイクル管理 (Maintenance & Lifecycle)

-   **鮮度維持**: 機能が実装されたら直ちに `FEATURES.md` のステータスを更新し、関連する `todos.md` のタスクを完了させてください。
-   **アーカイブ化**: 
    -   古くなった設計書や、廃止されたプロトコルの記録は、削除せず `docs/archive/` ディレクトリへ移動してください。
    -   アーカイブへ移動する際は、ファイルの冒頭に廃止日と理由を明記してください。
